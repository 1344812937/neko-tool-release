# Neko Tool

Neko Tool 是一个面向多节点协作场景的项目管理、目录浏览、差异比较与文件同步工作台。

它适合这样的场景：

- 在一台机器上统一登记多个本地项目目录
- 将多台运行 Neko Tool 的机器互相登记为远程节点
- 对同名项目或同类目录做差异比较
- 按文件或按整批差异项执行同步
- 在浏览和同步过程中保留文件级日志，便于回溯修改记录

项目当前采用 Go + Vue 3 单仓库实现，后端统一提供 API 和前端静态资源，默认通过单服务访问。

## 核心功能

### 1. 项目管理

- 维护本地项目清单，支持新增、重命名、删除项目
- 项目以“名称 + 编码 + 根目录”形式登记，便于后续比较和同步
- 支持两种目录选择方式：
  - 本机桌面环境下调用系统目录选择器
  - 远程访问场景下通过受限目录浏览器选择服务器目录
- 恢复已存在的项目记录后，会自动重新扫描缓存

### 2. 节点管理

- 登记远程 Neko Tool 节点
- 配置节点名称、服务根地址、共享令牌和说明信息
- 支持节点探测，确认远程服务是否可达
- 支持读取远程节点已暴露的项目列表

### 3. 项目浏览

- 以 Finder 分栏方式浏览项目目录
- 支持目录展开、文件详情查看、相对路径复制
- 文本文件可直接预览，非文本文件展示元数据摘要
- 浏览本机项目文件时，可查看该文件的历史修改记录和差异详情

### 4. 比较工作台

- 支持“本机节点 vs 本机节点”“本机节点 vs 远程节点”“远程节点 vs 远程节点”三类比较方式
- 支持限定相对目录范围；留空时比较整项目
- 差异结果按目录树展开，可定位到具体文件
- 文本文件支持 Monaco Diff 视图

### 5. 文件同步

- 支持单文件左 -> 右、右 -> 左同步
- 支持按当前比较结果执行整批同步
- 同步动作基于比较结果执行，适合在确认差异后再提交
- 同步后可回到浏览页或日志页继续核对结果

### 6. 站点信息与操作日志

- 首页展示本站名称、监听地址、认证状态、运行时长、共享令牌和当前版本
- 本站日志页汇总当前环境全部项目的文件操作记录
- 支持按项目名、文件路径、工作站、变更类型筛选日志
- 支持查看单条日志详情，并跳转到对应项目文件

## 页面说明

| 页面 | 入口路径 | 说明 |
| --- | --- | --- |
| 首页 | /static/ | 查看站点信息，并进入各工作台 |
| 认证页 | /static/auth | 输入访问令牌完成登录 |
| 项目管理 | /static/projects | 管理本地项目和目录访问能力 |
| 项目浏览 | /static/projects/:id/browser | 浏览目录、预览文件、查看文件修改记录 |
| 节点管理 | /static/nodes | 管理远程节点、探测连通性、读取远程项目 |
| 比较工作台 | /static/compare | 比较左右项目差异并执行同步 |
| 本站日志 | /static/site/logs | 查看当前环境所有项目操作日志 |

## 快速开始

### 环境要求

- Go 1.25+
- Node.js ^20.19.0 或 >= 22.12.0
- pnpm

### 1. 构建前端

后端会通过 go:embed 打包 frontend/dist，因此必须先构建前端。

```bash
cd frontend
pnpm install
pnpm build
```

### 2. 生成 Wire 代码（仅在依赖注入变更后需要）

```bash
cd cmd
go run -mod=mod github.com/google/wire/cmd/wire
```

### 3. 启动后端

在仓库根目录执行：

```bash
go run main.go
```

或构建二进制后运行：

```bash
go build -o neko-tool .
./neko-tool
```

如需跨平台构建，可执行：

```bash
sh build.sh
```

### 4. 打开页面

默认访问地址：

```text
http://localhost:8888/static/
```

根路径 / 会自动重定向到 /static/。

## 配置说明

默认配置文件位于 config/config.toml。首次运行时，如配置文件不存在，应用会先创建默认配置，再在当前命令行终端中引导填写关键配置项，写回完成后再继续启动。

如果现有配置文件缺少关键字段，应用也会在启动时补问缺失项；如果当前环境不是交互式终端，则不会等待输入，而是继续使用默认值或自动生成值启动，并把缺失字段回写到配置文件。

终端引导规则：

- 直接回车表示接受当前默认值
- node_config.workstation_address 允许留空，留空时应用会自动探测展示地址
- project_access.allowed_roots 支持连续输入多个目录，直接回车结束输入
- 非交互环境下不会弹出问答，适合 CI、后台服务或重定向输出场景

当前主要配置项如下：

### web_config

- host：监听主机地址
- port：监听端口

默认组合后即 <http://localhost:8888>。

### auth_config

- access_token：用户登录令牌

用途：

- 浏览器访问受保护页面时，需要先在认证页输入这个令牌
- 登录成功后，前端会保存签发后的认证凭据，后续请求自动携带

### node_config

- name：当前节点展示名称
- shared_token：节点间互相调用内部接口时使用的共享令牌

用途：

- 首页会显示当前节点名称和共享令牌
- 添加远程节点时，需要把对端节点的 shared_token 填到“共享令牌”字段

### project_access

- enable_native_local_picker：是否允许调用系统目录选择器
- allowed_roots：允许创建和浏览项目的根目录白名单
- follow_symlink：是否允许跟随符号链接

用途：

- 控制前端目录选择能力
- 约束项目只能落在允许访问的目录范围内
- 远程访问场景下，浏览器目录选择器也受 allowed_roots 限制
- 首次启动引导中，allowed_roots 可连续输入多个目录；直接回车结束，留空则表示远程访问时不提供白名单目录浏览

## 使用手册

### 第一步：启动服务并完成登录

1. 按“快速开始”启动服务。
2. 如果是首次启动，或配置文件缺少关键字段，先按命令行提示填写配置。
3. 配置写回后，应用会继续启动。
4. 打开 /static/ 页面。
5. 首次访问会跳转到认证页。
6. 输入 config/config.toml 中 auth_config.access_token 的值。
7. 登录成功后会回到原目标页面。

### 第二步：添加本地项目

1. 进入“项目管理”。
2. 点击“添加项目”。
3. 填写项目名称。
4. 填写项目编码。
   - 建议同一业务项目在不同节点使用同一编码，后续比较时更容易确认左右两侧是否为同一个项目。
5. 选择目录。
   - 本机访问且允许系统选择器时，可直接调用系统目录选择器。
   - 远程访问或禁用系统选择器时，可在允许的目录白名单中浏览服务器目录。
6. 保存后，项目会出现在项目列表中。

### 第三步：浏览项目与查看文件记录

1. 在“项目管理”列表中点击“浏览”。
2. 进入项目浏览页后，左侧按列展开目录结构。
3. 选择目录时可查看目录摘要。
4. 选择文件时：
   - 文本文件会在右侧显示内容预览
   - 二进制文件显示类型、摘要、大小等信息
5. 点击“修改 N 次”标签，可查看该文件的历史修改记录。
6. 打开单条记录后，可查看修改前后内容差异或对应元数据。

说明：

- 本机项目在浏览和刷新时会结合缓存与磁盘状态进行比对
- 当发现本地文件变化时，会补充对应日志记录，便于追踪后续同步结果

### 第四步：添加远程节点

1. 确保另一台机器也已启动 Neko Tool。
2. 在对端机器确认：
   - 服务访问地址，例如 <http://10.0.0.12:8888>
   - node_config.shared_token
3. 进入“节点管理”，点击“添加节点”。
4. 填写：
   - 节点名称：自定义识别名称
   - 节点地址：对端服务根地址
   - 共享令牌：对端节点当前的 shared_token
   - 说明：可选，用于标记环境或用途
5. 保存后，可点击“探测”验证连通性。
6. 如需确认对端项目是否已准备好，可点击“远程项目”查看对端项目清单。

### 第五步：执行项目比较

1. 进入“比较工作台”。
2. 分别选择左侧基准和右侧目标：
   - 节点
   - 项目
3. 如需只比较某个子目录，可填写“目录范围”；留空表示整项目。
4. 点击“执行比较”。
5. 比较结果会显示：
   - 总项数
   - 差异文件数
   - 差异目录数
   - 仅左侧
   - 仅右侧
6. 在差异分栏中选择具体目录或文件，查看详细内容。

差异类型说明：

- 仅左侧：只存在于左侧项目
- 仅右侧：只存在于右侧项目
- 文件变更：同路径文件内容不同
- 目录变更：目录层级存在结构变化
- 类型变更：同路径对象类型不同，例如一侧为目录，另一侧为文件

### 第六步：执行同步

Neko Tool 提供两类同步方式。

#### 1. 单文件同步

1. 在比较结果中选中某个文件。
2. 根据需要点击“左 -> 右”或“右 -> 左”。
3. 同步完成后，可再次刷新差异或直接查看日志。

#### 2. 整批同步

1. 在比较工作台执行完比较后。
2. 点击“整项目左 -> 右”或“整项目右 -> 左”。
3. 在弹窗中确认本次要执行的文件列表。
4. 提交后等待同步完成。

建议：

- 整批同步前先确认左右项目编码是否一致
- 对高风险目录先缩小到具体相对路径再操作
- 同步后及时到“本站日志”或“项目浏览”复核结果

### 第七步：查看本站日志

1. 进入“本站日志”。
2. 按项目名、文件路径、工作站、操作人 IP 或变更类型筛选。
3. 点击“查看详情”查看单条变更记录。
4. 点击“跳转项目”可直接打开对应项目浏览页并定位文件。

适合使用日志页的场景：

- 追踪某次同步是否真正落盘
- 回查某个文件最近是谁改过
- 核对本地基线、本地修改和同步覆盖记录

## 常见注意事项

### 1. 节点地址必须填写服务根地址

添加远程节点时，BaseURL 必须填写类似下面的地址：

```text
http://127.0.0.1:8888
```

不要填写：

- /static
- /static/#/projects
- 某个具体页面路径
- 不带协议的地址

否则远程节点可能返回 HTML 页面，导致节点探测、项目读取和同步全部失败。

### 2. 系统目录选择器依赖桌面环境

本机目录选择功能依赖系统文件选择器。无桌面环境、远程 SSH 会话或服务器场景下，这个能力可能不可用。

这时应改用浏览器目录选择方式，并提前在 project_access.allowed_roots 中配置允许访问的目录。

### 3. allowed_roots 会限制项目可访问范围

如果某个目录无法选择、无法创建项目或无法浏览，优先检查：

- project_access.allowed_roots 是否已包含目标根目录
- 当前访问方式是否允许本机目录选择器
- 当前路径是否超出服务器允许的边界

### 4. 共享令牌必须与对端一致

节点之间调用内部接口时会校验 shared_token。

如果出现远程节点鉴权失败，优先检查：

- 本地保存的共享令牌是否就是对端 config/config.toml 中的 node_config.shared_token
- 对端节点是否启用、是否已重启加载最新配置
- 双方系统时间是否严重不一致

### 5. 前端基路径固定为 /static/

当前前端路由和后端静态资源服务都基于 /static/。

因此：

- 页面访问入口应使用 /static/
- 如果你修改部署方式或代理规则，需要同时考虑静态资源路径与前端路由基路径

### 6. 后端启动前必须先构建前端

因为后端会嵌入 frontend/dist，如果前端未构建：

- 后端可能无法构建
- 或继续提供旧版前端资源

## 常见排查思路

### 登录失败

- 检查 auth_config.access_token 是否输入正确
- 检查是否访问了正确的服务实例

### 节点探测失败

- 检查节点地址是否可直接在浏览器中访问到服务根地址
- 检查协议是否正确，是 http 还是 https
- 检查目标机器防火墙、路由、VPN 或 DNS 配置
- 检查 shared_token 是否与对端一致

### 远程节点返回页面内容而不是接口结果

这通常表示节点地址填成了页面路径，而不是服务根地址。应改为：

```text
http://host:8888
```

而不是：

```text
http://host:8888/static/
```

### 无法读取远程项目

- 检查对端是否已经添加本地项目
- 检查对端 allowed_roots 是否允许访问目标目录
- 检查对端服务是否正常启动

## 开发补充

如果你需要继续开发这个项目，建议先了解下面几点：

- 这是一个 Go + Vue 3 单仓库
- 前端位于 frontend/
- 后端主代码位于 internal/
- 依赖注入入口位于 cmd/wire.go
- 后端通过 main.go 的 go:embed 打包前端产物

常用开发命令：

```bash
# 前端开发
cd frontend && pnpm dev

# 前端构建
cd frontend && pnpm build

# Wire 生成
cd cmd && go run -mod=mod github.com/google/wire/cmd/wire

# 后端运行
go run main.go
```

更细的后端分层规范和约束说明，请继续阅读 CLAUDE.md、pkg/core/tx/README.md、pkg/repository/README.md 和 pkg/service/README.md。
